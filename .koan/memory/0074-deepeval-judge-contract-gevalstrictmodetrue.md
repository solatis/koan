This entry documents the judge-LLM contract for the DeepEval-based koan eval harness in `evals/scorers.py`. Five dated revisions shape the current state.

On 2026-04-22, Leon chose `GEval(name=..., criteria=<rubric-body>, strict_mode=True, model=JUDGE_MODEL)` for per-phase rubrics, with `JUDGE_MODEL = GeminiModel(model="gemini-3-pro")` as a shared module constant. Rationale: binary PASS/FAIL matched the rubric files' existing "PASS or FAIL on the last line" tail contract, and passing the full rubric body via `criteria=` preserved the migration's "don't change rubric content" constraint. `DAGMetric` for per-criterion decomposition was deferred. `LiteLLMModel('google/gemini-3.1-pro-preview')` was rejected (plan-review caught the nonexistent `3.1` family).

On 2026-04-23 morning, during an intake-phase redesign, Leon adopted DAGMetric for a new `RubricCompliance` dimensional metric -- each rubric bullet becoming a `BinaryJudgementNode` aggregated via VerdictNodes.

On 2026-04-23 afternoon, plan-review reversed the DAGMetric choice based on a reading of the root-node constraint. `DeepAcyclicGraph.__init__` in deepeval 3.9.7 raises `ValueError("You cannot provide more than one root node when using 'BinaryJudgementNode' or 'NonBinaryJudgementNode' in root_nodes.")` whenever more than one binary judgement appears as root. The replacement was a custom `RubricComplianceMetric(BaseMetric)` that takes no criteria at construction, reads `rubric_criteria` from `LLMTestCase.additional_metadata` inside `a_measure()`, and issued one `JUDGE_MODEL.a_generate_with_schema(prompt, CriterionVerdict)` call per criterion via `asyncio.gather`. `strict_mode=True` binary semantics preserved. `CrossPhaseCoherenceMetric(BaseMetric)` replaced GEval similarly, reading `rubric_body` from `additional_metadata`. `JUDGE_MODEL` upgraded to `GeminiModel(model="gemini-3-pro-preview")` in the same revision.

On 2026-04-23 late, two findings reopened the DAGMetric direction. First, the Confident AI dashboard renders "No verbose logs available" for every custom BaseMetric row because its frontend carries a CSS class `disableVerbose` that hides verbose content for non-DeepEval-authored metrics; DAGMetric's `construct_node_verbose_log()` is the only dashboard-observable path. Second, the `asyncio.gather` over criteria x `AsyncConfig(max_concurrent=100)` at evaluate() level produced a 136+ concurrent Gemini flood; that was retired in favor of sequential per-criterion iteration and `max_concurrent=10`. Leon's planned direction on 2026-04-23 late: refactor `RubricComplianceMetric` to a DAGMetric with one `BinaryJudgementNode` per rubric criterion rooted under a single `TaskNode` to satisfy the single-root constraint.

On 2026-04-24, during plan-spec for the DAGMetric refactor, Leon rejected the TaskNode-root approach after adversarial review identified two disqualifying issues. First, `TaskNode._a_execute` in `deepeval/metrics/dag/nodes.py` runs its children via `asyncio.gather`, reintroducing the thundering-herd problem (memory entry 97) at the per-criterion level: for an 11-criterion rubric x 10 concurrent cases, peak Gemini concurrency returns to ~110. Second, `DAGMetric` holds a single scalar `metric.score` that every `VerdictNode._a_execute` overwrites via `metric.score = self.score / 10`; when N BinaryJudgementNodes run in parallel under a TaskNode root, the final score is set by whichever VerdictNode finishes last -- non-deterministic, not a pass-rate. Also rejected: chaining BJNs via `VerdictNode.child` (short-circuits on first failure, losing verbose logs for un-reached criteria). The adopted resolution was one DAGMetric per criterion, each with a single `BinaryJudgementNode` root and two `VerdictNode` leaves (`VerdictNode(verdict=True, score=10)` / `VerdictNode(verdict=False, score=0)`). Criteria compose at the metric-list level via `assert_test(tc, [metric_1, ..., metric_N])` rather than the DAG level, sidestepping the root-node constraint entirely. Factory functions `build_criterion_metric(criterion, name) -> DAGMetric` and `build_cross_phase_metric(rubric_body, name) -> DAGMetric` in `evals/scorers.py` replaced the construction-parameterless BaseMetric singletons. `CrossPhaseCoherenceMetric` became one DAGMetric with a single BJN containing the full rubric body as `criteria=` (per task description: one holistic judgment, not per-criterion). Metric naming distinguishes fixture-level criteria `Fixture_{phase}_{section}_{idx:02d}` (shared across tasks) from task-addendum criteria `Task_{task_id}_{phase}_{section}_{idx:02d}` (task-scoped) so Confident AI dashboard group-by-metric yields meaningful cross-run comparison.
