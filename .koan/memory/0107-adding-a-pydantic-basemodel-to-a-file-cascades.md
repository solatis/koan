title: Adding a Pydantic BaseModel to a file cascades nested-model validation strictness
  onto sibling models
type: lesson
created: '2026-04-24T16:39:51Z'
modified: '2026-04-24T16:39:51Z'
---

This entry records a surprise surfaced during the file-attachment initiative's M5 execute phase in `koan/projections.py`. On 2026-04-24, the executor added an `AttachmentEntry(KoanBaseModel)` class inside `koan/projections.py` to represent the audit-log attachment manifest emitted on `tool_completed` events. `koan/projections.py` already contained many sibling `KoanBaseModel` subclasses including `Proposal`. Immediately after the `AttachmentEntry` class was added, Pydantic validation that runs when folding `memory_curation_started` events began raising `ValidationError` on `Proposal.seq` and `Proposal.meta` -- fields that had accepted non-string values from upstream call sites for weeks without issue. Root cause identified during exec-review: Pydantic's nested-BaseModel validation is stricter than validation on sibling top-level models in the same module; adding a referenced BaseModel can cascade that strictness onto the parent class's other fields under some validation pathways. The existing dormant edge case (upstream producing dicts or non-string values for `seq` / `meta`) was always a latent bug but had not been exercised. Fix applied on 2026-04-24 in the same change: wrap the affected fields in `Annotated[str, BeforeValidator(_coerce_str)]` with a coercion function that accepts `str` as-is, `json.dumps`-serializes `dict`, and falls back to `str(v)` for everything else. Defaults set to `""`. Lesson for future additions of Pydantic BaseModel classes to files with existing Pydantic models in koan: run the full non-eval test suite (`uv run pytest --ignore=tests/evals`) immediately after the add, not after the surrounding change is otherwise complete -- dormant validation gaps surface as test failures, and the fix is usually a `BeforeValidator` wrapper rather than a deeper redesign.
