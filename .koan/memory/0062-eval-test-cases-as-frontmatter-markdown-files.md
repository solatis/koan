The koan-bench Inspect AI test-case model in `evals/` was introduced on 2026-04-19 during a reorganization that split fixture state from test-case definition. On 2026-04-19, Leon decided that each test case is a single markdown file at `evals/fixtures/<fixture>/tasks/<task>/cases/<slug>.md` with YAML frontmatter declaring `workflow` (string, e.g. `"plan"`) and `directed_phases` (a list ending in `"done"`). The body below the closing `---` is the cross-cutting rubric graded by the `workflow_overall` scorer. Leon stated the reframe: "the rubrics are actually test cases, but separately we want to execute the workflow in a few directed ways." Cases are parsed by a new `evals/cases.py` module using `yaml.safe_load` (no new dependency -- `pyyaml` was already in `pyproject.toml`).

Leon decided that `evals/tasks.py` synthesizes one Inspect `@task` per discovered case at module import time by writing into `globals()`, with function names of the form `koan_<fixture>_<task>_<case>` and hyphens converted to underscores for Python-identifier compliance. A `_make_task(case)` helper creates the closure so each factory captures its own `case` variable, avoiding Python's late-binding loop-variable pitfall. Each factory wires only the scorers for phases in its case's `directed_phases`; per-phase scorers return `None` (Inspect skips them) when the scorer's phase is not in the Sample's declared `directed_phases` after dropping `"done"`. The `workflow_overall` scorer loads its rubric from the case body (stored in Sample metadata under `overall_rubric`), not from a separate file.

Correction (2026-04-19, later that day): the original verification claim in this memory -- that `uv run inspect eval evals/tasks.py --list` confirmed discovery of the dynamically-registered factories -- was wrong on two counts. First, `--list` is not a real `inspect eval` flag (the CLI errors with "No such option: --list"). Second, and more important, Inspect AI's CLI task discovery is strictly AST-based: `inspect_ai/_util/decorator.py::parse_decorators` + `inspect_ai/_eval/loader.py::code_has_task` scan the source file for a top-level `@task`-decorated `FunctionDef` before `load_module` is even allowed to run. A file whose only `@task` applications happen via dynamic `task(_factory)` calls inside a loop is rejected at the AST gate, `exec_module` never runs, and the dynamic factories never get registered. The original verification was done via `python -c "import evals.tasks"` (direct module import bypasses the AST gate), which is not the path Inspect's CLI uses. The CLI failure mode is `Error: No inspect tasks were found at the specified paths.`

The fix, applied on the same day, was to add one real static `@task def koan()` to `evals/tasks.py` that satisfies the AST gate. Once the file passes `code_has_task`, `load_module` calls `exec_module`, which runs the `for _case in discover_cases(...)` loop and registers every dynamic factory in Inspect's task registry. Subsequent resolution of `inspect eval evals/tasks.py@koan_<fixture>_<task>_<case>` succeeds because `create_tasks` calls `load_file_tasks(path)` (which triggers the exec) before `task_create(name)` looks the dynamic factory up in the registry. The static `koan` task doubles as a run-all suite entry point: it returns a `Task` over the full `load_dataset()`, attaches all nine scorer factories, and relies on the per-sample `directed_phases` gate inside `_rubric_scorer` for per-sample filtering.

Rejected alternatives on 2026-04-19 during intake: JSON sidecar files next to each case (Leon called them "ugly"); manual per-case registry in `tasks.py` (Leon rejected it for breaking data-driven discovery); explicit `phases_scored` frontmatter field (Leon rejected as duplicating information already in `directed_phases`); flat `evals/cases/*.md` tree with fixture+task in frontmatter (Leon rejected for weaker cohesion with task state); nested case directories under a separate top-level `cases/` tree mirroring `fixtures/` (Leon rejected in favor of co-locating cases with task state under `tasks/<t>/cases/`).

One previously-listed rejected alternative -- "one `@task` over a multi-Sample dataset" -- has been reinstated as complementary, not exclusive: the static `koan` task added for the AST-gate fix IS exactly that shape (one `@task`, dataset is every case as a Sample, per-sample scoring filtered via `directed_phases` metadata). Per-variant addressability is preserved via the dynamic `koan_<fixture>_<task>_<case>` factories, which coexist with the static one. The final shape is "both": a static suite entry point plus per-case individually-named factories, both registered at module import time.
